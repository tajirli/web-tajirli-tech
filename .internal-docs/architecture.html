System Architecture

Layered architecture with a migration bridge: legacy JS APIs are kept stable while new typed services are introduced in feature modules.

Overview Architecture Modules + API Progress Tracker Runbook

Layer Responsibilities

Layer Main Files Responsibility Status
Entry + Shell src/main.jsx, src/App.jsx Bootstraps providers, routing, lazy pages, directionality, theme variables Stable
UI + Pages src/components/*, src/pages/* Storefront screens, admin screens, conversion components Stable
State + Context src/context/SiteSettingsContext.jsx, src/context/LayoutContext.jsx Theme, branding, tracking IDs, shell visibility, local persistence Active changes
Domain Features src/features/products/*, src/features/orders/* Typed business services and hooks for domain operations In migration
API Bridge src/api.js Backward-compatible exports for existing JS components Temporary bridge
Core Infrastructure src/core/api/httpClient.ts, src/core/config/app.config.ts HTTP abstraction, retry policy, app constants Stable

Cross-Cutting

  • i18n language auto-detection and RTL/LTR switching
  • Dynamic pixel injection (Meta, Snapchat, TikTok)
  • Visitor telemetry batching to /api/public/track
  • Admin session token persistence in localStorage
  • Product cache warm-up and TTL cache strategy

Data/Request Flow

User action in page/component
|
Hook/context triggers domain or API bridge function
|
Axios client with retry/backoff performs request
|
Backend returns response or error
|
UI updates state + toast feedback + optional analytics event

Architecture Risks To Monitor

  • Dual API paths (legacy bridge + new services) can diverge.
  • Long i18n resource file increases merge conflict probability.
  • Pixel script injection errors can break tracking silently.
  • Client-side admin token storage needs strict browser hygiene.
  • Fallback network retries may mask backend availability issues.
